# rag_app_v7.py 변동사항 요약

## 핵심 변경: 멀티문서 Reasoning (3단계 추론 파이프라인)

v6의 단일 LLM 호출 답변 생성을 3단계 추론 파이프라인으로 교체. 청크를 바로 LLM에 던지지 않고, 요약 → 관계 분석 → 최종 답변 순서로 단계적 추론.

---

## 최종 스택
| 구성 | v6 (rag_app_v6.py) | v7 (rag_app_v7.py) |
|------|---------------------|---------------------|
| 임베딩 | text-embedding-3-small | text-embedding-3-small (동일) |
| LLM | gpt-4o-mini | gpt-4o-mini (동일) |
| 벡터 DB | FAISS - 코사인 유사도 | FAISS - 코사인 유사도 (동일) |
| 청킹 | 문단/문장+Overlap / Semantic+Overlap | 동일 |
| 쿼리 처리 | 리라이팅 + 멀티쿼리 | 동일 |
| 리랭킹 | LLM 배치 리랭킹 | 동일 |
| 답변 구조 | 요약→근거→결론 (단일 LLM 호출) | **3단계 추론 파이프라인** |
| 출처 추적 | 없음 | **파일명별 chunk_sources 추적** |

```python
    # → 정규식으로 파싱 → 리스트로 반환
```
**이유:** 리랭킹된 청크들이 각각 긴 원문 텍스트이므로, LLM이 한꺼번에 처리하면 중요한 정보를 희석시킬 수 있음. 질문 관점의 압축 요약으로 Step2/3의 입력 품질을 높임.

**이유:** 단일 문서 RAG에서는 불필요하지만, 여러 문서를 업로드했을 때 각 문서 간 정보가 일치하는지 충돌하는지를 LLM이 명시적으로 판단하게 함. 최종 답변에서 모순된 정보를 무시하거나 혼합하는 오류 방지.

---

### 4. 파일명별 출처 추적 (`chunk_sources`)
청크 처리 시 파일명을 함께 저장해 청크가 어느 파일에서 왔는지 추적.
```python
# 변경 전: 청크 텍스트만 저장
st.session_state.chunks = all_chunks

# 변경 후: 청크와 파일명을 병렬 리스트로 저장
st.session_state.chunks = all_chunks
st.session_state.chunk_sources = all_sources  # ["doc1.pdf", "doc1.pdf", "doc2.txt", ...]
```
**이유:** 여러 문서를 업로드했을 때 답변의 [출처 N]이 어느 파일에서 왔는지 명확히 표시. Step2 분석에서도 파일명을 함께 전달해 문서 간 비교가 더 명확해짐.


## 추론 파이프라인 전체 흐름
```
[v6] 질문 → 검색 → 리랭킹 → LLM 1회 호출 → 답변

[v7] 질문
  → 쿼리 리라이팅 (검색 Recall ↑)
  → 멀티쿼리 검색
  → LLM 리랭킹 (Precision ↑)
  → [Step 1] 청크별 요약 (LLM 1회, 배치)
  → [Step 2] 공통점·차이점 분석 (LLM 1회)
  → [Step 3] 최종 구조화 답변 (LLM 1회)
  ※ 추가 LLM 호출 2회 (Step1·2), 대신 답변 품질·근거 추적성 대폭 향상
```

## 성능 개선 기대 포인트
- 다중 문서 간 정보 충돌 감지: Step2에서 명시적으로 차이점을 찾아 최종 답변에 반영
- 근거 추적성 강화: 파일명까지 표시되어 어느 문서의 어느 청크가 근거인지 확인 가능
- 단계적 추론으로 Hallucination 억제: 원문을 바로 LLM에 던지지 않고 요약·분석을 거쳐 최종 답변 생성
